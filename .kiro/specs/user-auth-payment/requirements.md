# Requirements Document

## Introduction

Добавление системы аккаунтов и монетизации в Telegram Mini App «Калорийный трекер».
Приложение работает на Next.js (Vercel), данные хранятся в localStorage через Zustand.
Авторизация — через Telegram WebApp API (без отдельного логина/пароля).
Монетизация — разовый платёж 50 рублей через FreeKassa, открывающий пожизненный доступ.
Бэкенд и база данных (Supabase) создаются в рамках этой фичи.

## Glossary

- **Auth_Service**: серверный модуль (Next.js API Route), отвечающий за регистрацию и верификацию пользователей через Telegram
- **Payment_Service**: серверный модуль, отвечающий за создание платёжных ссылок FreeKassa и обработку webhook-уведомлений об оплате
- **Access_Guard**: клиентский компонент, проверяющий статус доступа пользователя и блокирующий функционал при необходимости
- **User**: запись в базе данных Supabase, идентифицируемая по `telegram_id`
- **Trial_Period**: 3-дневный бесплатный период с момента первой регистрации
- **Subscription**: статус пожизненного доступа, активируемый после успешной оплаты
- **telegram_id**: уникальный числовой идентификатор пользователя Telegram, получаемый из `Telegram.WebApp.initDataUnsafe.user.id`
- **initData**: строка инициализации Telegram WebApp, передаваемая в заголовке запроса для верификации на сервере
- **FreeKassa**: платёжная система для приёма платежей на карту владельца

## Requirements

### Requirement 1: Авторизация через Telegram

**User Story:** Как пользователь Telegram Mini App, я хочу автоматически авторизоваться через свой Telegram-аккаунт, чтобы не вводить логин и пароль.

#### Acceptance Criteria

1. WHEN приложение загружается в Telegram, THE Auth_Service SHALL извлечь `telegram_id` и данные пользователя из `Telegram.WebApp.initDataUnsafe`
2. WHEN пользователь открывает приложение впервые, THE Auth_Service SHALL создать новую запись User в базе данных Supabase с `telegram_id`, именем пользователя и датой регистрации
3. WHEN пользователь открывает приложение повторно, THE Auth_Service SHALL найти существующую запись User по `telegram_id` и вернуть текущий статус доступа
4. WHEN запрос к Auth_Service поступает без валидного `initData`, THEN THE Auth_Service SHALL отклонить запрос с кодом ошибки 401
5. THE Auth_Service SHALL верифицировать подлинность `initData` с использованием HMAC-SHA256 и секретного ключа бота

### Requirement 2: Бесплатный пробный период

**User Story:** Как новый пользователь, я хочу получить 3 дня бесплатного доступа ко всему функционалу, чтобы оценить приложение перед оплатой.

#### Acceptance Criteria

1. WHEN новый User создаётся в базе данных, THE Auth_Service SHALL записать `trial_started_at` равным текущему времени UTC
2. WHILE `trial_started_at` установлен и прошло менее 72 часов с момента `trial_started_at`, THE Access_Guard SHALL предоставить пользователю полный доступ ко всему функционалу
3. WHEN прошло 72 или более часов с момента `trial_started_at` и Subscription отсутствует, THE Access_Guard SHALL заблокировать доступ к функционалу и отобразить экран оплаты
4. THE Access_Guard SHALL отображать пользователю оставшееся время пробного периода в виде количества дней и часов

### Requirement 3: Согласие с условиями при регистрации

**User Story:** Как владелец приложения, я хочу, чтобы новый пользователь явно согласился с условиями использования, чтобы он понимал, что через 3 дня потребуется оплата.

#### Acceptance Criteria

1. WHEN пользователь открывает приложение впервые, THE Access_Guard SHALL отобразить экран согласия с условиями до предоставления доступа к функционалу
2. THE Access_Guard SHALL отображать на экране согласия текст, явно указывающий, что пробный период составляет 3 дня, после чего требуется разовая оплата 50 рублей для пожизненного доступа
3. WHEN пользователь нажимает кнопку подтверждения согласия, THE Auth_Service SHALL записать `terms_accepted_at` и `trial_started_at` в запись User и начать пробный период
4. WHILE пользователь не подтвердил согласие с условиями, THE Access_Guard SHALL не предоставлять доступ к функционалу приложения
5. IF пользователь закрывает экран согласия без подтверждения, THEN THE Access_Guard SHALL повторно отобразить экран согласия при следующем открытии приложения

### Requirement 4: Блокировка функционала после окончания пробного периода

**User Story:** Как владелец приложения, я хочу, чтобы после окончания пробного периода функционал был заблокирован до оплаты, чтобы стимулировать монетизацию.

#### Acceptance Criteria

1. WHEN пробный период истёк и Subscription отсутствует, THE Access_Guard SHALL заменить содержимое всех защищённых страниц на экран с предложением оплаты
2. WHEN пробный период истёк и Subscription отсутствует, THE Access_Guard SHALL отображать сообщение об истечении пробного периода и кнопку перехода к оплате
3. WHILE Subscription активна, THE Access_Guard SHALL предоставлять полный доступ ко всему функционалу без ограничений
4. THE Access_Guard SHALL проверять статус доступа при каждой загрузке страницы приложения

### Requirement 5: Оплата через FreeKassa

**User Story:** Как пользователь с истёкшим пробным периодом, я хочу оплатить 50 рублей и получить пожизненный доступ, чтобы продолжить использование приложения.

#### Acceptance Criteria

1. WHEN пользователь нажимает кнопку оплаты, THE Payment_Service SHALL сформировать уникальную платёжную ссылку FreeKassa для данного User с суммой 50 рублей
2. WHEN платёжная ссылка сформирована, THE Access_Guard SHALL перенаправить пользователя на страницу оплаты FreeKassa
3. WHEN FreeKassa отправляет webhook-уведомление об успешной оплате, THE Payment_Service SHALL верифицировать подпись уведомления и обновить статус User, установив `subscription_activated_at`
4. WHEN `subscription_activated_at` установлен, THE Access_Guard SHALL немедленно предоставить пользователю полный доступ ко всему функционалу
5. IF FreeKassa отправляет webhook-уведомление с невалидной подписью, THEN THE Payment_Service SHALL отклонить уведомление и не изменять статус User
6. THE Payment_Service SHALL использовать уникальный `order_id` для каждого платёжного запроса, чтобы исключить повторную активацию по одному платежу

### Requirement 6: Синхронизация данных пользователя

**User Story:** Как пользователь, я хочу, чтобы мои данные (дневник, настройки) сохранялись в облаке, чтобы не терять их при смене устройства.

#### Acceptance Criteria

1. WHEN пользователь авторизован и имеет активный доступ, THE Auth_Service SHALL связать данные localStorage с записью User в Supabase по `telegram_id`
2. WHEN пользователь открывает приложение на новом устройстве, THE Auth_Service SHALL загрузить данные пользователя из Supabase и синхронизировать их с localStorage
3. WHEN пользователь вносит изменения в дневник питания или настройки, THE Auth_Service SHALL сохранить изменения в Supabase в течение 5 секунд
4. IF соединение с Supabase недоступно, THEN THE Auth_Service SHALL продолжить работу с локальными данными и выполнить синхронизацию при восстановлении соединения
